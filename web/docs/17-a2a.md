# A2A Agents

**Agent-to-Agent (A2A)** is an open protocol for routing tasks between
independent agent servers. Yoke can act as both a client (calling remote
A2A endpoints) and a server (receiving inbound calls from other agents).

The **Settings → A2A** panel manages outbound connections — the remote
endpoints the leader can delegate to.

## Config file

Connections are stored in `config/a2a_config.json`:

```json
{
  "agents": {
    "peer-yoke": {
      "url": "http://peer-host:8091/",
      "description": "Secondary yoke server.",
      "headers": { "Authorization": "Bearer ${input:peer_token}" },
      "squad": "",
      "session_name": "",
      "create": false
    }
  },
  "inputs": [
    { "id": "peer_token", "type": "promptString", "description": "Peer bearer token", "password": true }
  ]
}
```

The Web UI editor reads from the config search chain but always writes
to `$HOME/.yoke/config/a2a_config.json`.

## Panel sub-tabs

### Agents sub-tab

Each configured peer appears as a card. The **Add A2A Agent** card at the
end creates a new blank entry.

Each card has four sections:

**General Settings**

- **Name** — the key used to reference this peer. Must be unique. The
  leader sees it as a tool named `a2a_<name>`.

**Connection**

- **URL** — full base URL of the remote A2A server (e.g.
  `https://agent.example.com/`).
- **Description** — optional free-text description surfaced in the
  leader's tool catalogue.

**Routing** — defaults used when the leader invokes this peer. The agent
can override them per call.

- **Squad** — which squad to run on the remote server (blank = remote's
  `default`).
- **Session Name** — route into a persistent named session on the remote.
  Leave blank for an ephemeral per-task session that is discarded after
  the response.
- **Create session if missing** — when set, a named session that does not
  yet exist is created automatically (mirrors `POST /api/sessions` with a
  name on the remote side).

**Headers**

Arbitrary HTTP headers sent with every request to this peer. Common use:

```
Authorization   Bearer ${input:peer_token}
```

Header values support `${input:id}` substitution — the value is replaced
at call time by the corresponding **Input** (see below), so credentials
stay out of the saved config.

### Remotes sub-tab

Browse and install A2A agent definitions from remote registries. Uses the
same registry infrastructure as skills and agents — see the remote
registries section in the Agents documentation.

## Inputs

**Inputs** are named secrets declared once and referenced from header
values. At call time the agent emits an `ASK_USER` sentinel; the Web UI
prompts the user, caches the answer for the rest of the session, and
coalesces concurrent requests for the same input. The answer is never
persisted to disk.

Input types:

| Type | Behaviour |
|---|---|
| `promptString` | Free-text prompt. Set `password: true` to mask input. |
| `pickString`   | Dropdown from a fixed `options` list. |

## Wiring a peer to an agent

After adding a peer in the A2A panel, open **Settings → Agents**, select
the leader (or any agent that should be able to delegate), and enable the
peer under the **A2A Agents** block. Only enabled peers appear as tools
on that agent.

## Receiving inbound calls (A2A server)

To accept calls from other A2A agents, enable the A2A server in
`config/server.yaml`:

```yaml
a2a_enabled: true
a2a_port: 8091
```

The server handles `tasks/send` and `tasks/sendSubscribe`. Inbound
`metadata.squad` and `metadata.session_name` fields follow the same
routing rules as the outbound **Routing** section above.
